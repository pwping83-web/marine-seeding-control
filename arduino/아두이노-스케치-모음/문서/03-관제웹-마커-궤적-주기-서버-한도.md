# 03. 관제 웹 — 마커가 어떻게 찍히고, 몇 초마다 갱신되는지

아두이노가 **지도에 직접 그리지 않습니다.**  
**서버에 저장**되고, **관제 웹(브라우저)** 가 주기적으로 읽어서 **마커·선**을 그립니다.

---

## 1. 데이터 두 종류 = 화면에 두 가지 표현

### A) 종자 **살포 1건** (이벤트)

- **아두이노가 하는 일**: 살포 순간(또는 버튼)마다 `telemetry-ingest` 로 JSON 전송.  
- **서버**: `seed_drop_records` 테이블에 한 줄 저장.  
- **웹 지도**: **원형 마커(동그라미)** 여러 개 + 라벨.  
- **한 번 전송 = 한 개(또는 묶음)의 살포 기록** 으로 이해하면 됩니다.

### B) **선박 궤적** (항해 중 위치 줄)

- **아두이노가 하는 일**: **정해진 주기**(예: 30초~2분)마다 `vessel-track-ingest` 로 좌표 전송.  
- **서버**: `vessel_track_points` 테이블에 시간 순으로 쌓임.  
- **웹 지도**:  
  - **주황색 실선** = LTE로 쌓인 궤적  
  - **선박 아이콘** = 가장 최근 위치(테스트 모드에서 LTE “살아 있음”일 때)

---

## 2. “몇 초마다” 찍히나 — **두 가지 시계**가 있음

| 구분 | 누가 | 대략 값 (이 저장소 기본) |
|------|------|---------------------------|
| **기기 → 서버** | 아두이노가 POST 보내는 간격 | **스케치에서 정함** (예: 60초). 운영 정책·요금에 맞게 조절. |
| **서버 → 화면 (선박 궤적)** | 웹이 `vessel_track_points` 를 **다시 읽는** 간격 | 기본 **약 12초** (`VITE_VESSEL_LTE_POLL_MS` 로 5초~120초) |
| **서버 → 화면 (살포 마커)** | 웹이 `seed_drop_records` 를 **다시 읽는** 간격 | 기본 **약 12초** (`VITE_SEED_DROP_POLL_MS` 로 4초~120초). 아두이노·Edge로 들어온 점이 **새로고침 없이** 지도에 반영됨 |

**중요**: 기기를 **1초마다**내도, 웹은 **12초마다** 그릴 수 있어서, 화면 갱신은 **둘 중 느린 쪽 + 둘의 조합**으로 보입니다.  
살포 마커는 **전송 직후 다음 폴링 때** 반영되거나, 대시보드가 시드 목록을 다시 불러오는 타이밍에 따라 약간 지연될 수 있습니다.

---

## 3. 서버가 막는 속도 (남용·과부하 방지)

Edge 함수에서 **대략** 다음과 같이 제한합니다 (코드 기준, 변경 시 문서도 수정).

| API | IP당 초당 | 비고 |
|-----|------------|------|
| `telemetry-ingest` (살포) | 약 25건/초 | 한 요청에 `records` 최대 80개 |
| `vessel-track-ingest` (궤적) | 약 15건/초 | `points` 최대 100개 |

**현장 권장**: 살포는 **이벤트 때만**, 궤적은 **30~120초** 주기가면 충분한 경우가 많습니다. (불필요하게 1초마다 보내지 않기)

---

## 4. “신선도” 표시 (웹 UI)

- 궤적 토글 켠 상태에서 **최근 LTE 점이 25분보다 오래되면** “신호 없음” 류 안내가 뜨도록 되어 있습니다.  
- 이 값은 **운영 정책**에 맞게 나중에 코드에서 조정할 수 있습니다.

---

## 5. 정리 표 (비전문가용)

| 질문 | 답 |
|------|-----|
| 마커는 누가 찍나요? | **웹**이 DB를 읽어서 찍습니다. |
| 아두이노는? | **숫자만 보냅니다.** |
| 몇 초마다 움직여 보이나요? | **보내는 주기**와 **웹 폴링 주기** 둘 다 영향. 기본 폴링은 **약 12초**. |
| 살포는? | **버튼 누를 때마다**내면 됩니다. |
