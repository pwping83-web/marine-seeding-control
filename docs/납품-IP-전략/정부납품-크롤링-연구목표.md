# 정부 납품·판매 연구 목표 (본 프로젝트)

## 목표 한 줄

**해상 종자살포 관제(단말·통신·웹 대시보드)** 를 공공 수요와 맞추고, **나라장터 등 공고 정보를 자동 수집·알림**해 입찰·과제 기회를 놓치지 않는다.

## 공고·지원 검색 범위 (좁히지 않기)

공고·지원사업·입찰을 찾을 때 **「해양 종자 살포」만** 기준으로 두면 관련 수요를 놓친다. 동일한 기술 스택(위치·지도·통신·현장 단말·대시보드)으로 묶어 **넓게** 본다.

- **해도·해양 지도 데이터**: 전자해도·수심·해양공간정보·GIS/공간정보 고도화, 국제 표준(ENC·S-100 계열) 연계, 민·관 해도·지도 유지보수·구축 용역  
- **해저·해상 시설의 위치·표기**: 해저 케이블·관로 등 **선로·시설 위치**의 지도·정보체계 반영, 통항·안전·작업 구역 표시, 선박·부표·구조물과의 공간 일치  
- **현장·관제 일반**: 해상 모니터링, VTS·해양안전, 스마트해양, IoT·LTE·위성, 데이터 플랫폼, SW·정보시스템 유지보수  

위 축은 **제안서·검색 키워드**에서 종자·살포 키워드와 **병행**한다(종자만으로 필터하지 않음).

## 납품·판매 채널 (연구 대상)

1. **나라장터** 물품·용역·공사 입찰, 지자체 연계 입찰
2. **부처·공공기관 R&D·사업공모** (해양수산, 스마트해양, 수산종자·양식, 안전·환경 등 키워드 병행 검색)
3. **지식재산** (특허·실용신안·상표·디자인) 을 납품 실적·제안서의 **차별화·신뢰** 근거로 연결

나라장터·조달이 아닌 **행안부·타 부처 공지, 민원·이메일 루트**는 `공공정보수집-및-문의-루트.md` 를 본다.

## 크롤링·알림 원칙

- **웹 크롤링**보다 **공공데이터포털 OpenAPI**(조달 입찰공고 등)를 우선 사용한다. (약관·안정성)
- 레포의 `scripts/g2b-bid-watch.mjs` 로 **용역·물품·공사** 입찰공고 목록 API를 주기 조회하고, 웹훅·ntfy로 알린다. (`G2B_BID_TYPES` 로 종류 선택)
- **Windows** 에서 **3일마다 오후 3시경** 실행: `scripts/Register-G2bWatchScheduledTask.ps1` 로 작업 스케줄러 등록(매일 15시 호출, 래퍼가 3일 간격만 실제 조회).
- “딱 맞는 정보”를 위해 `G2B_KEYWORDS` 에 다음 유형을 조합해 넣는다 (예시, 필요 시 로컬 메모에 정리한 출원·사업명과 맞출 것). **종자·살포만 쓰지 말고** 해도·지도·공간정보·해저시설 쪽을 섞는다:
  - 해양, 수산, 양식, 종자, 살포, 관제, 모니터링, IoT, LTE, 위치, 선박, GIS, 공간정보, 클라우드, 소프트웨어, 정보시스템, 용역, 유지보수
  - 해도, 전자해도, 수심, 해양공간정보, ENC, 항로, 통항, 해저, 케이블, 관로, 지도, 영상, 측량, S-100

## 확장 아이디어 (에이전트가 별도 허락 없이 제안·구현 후보)

- 동일 API 패밀리에서 **물품·공사** 오퍼레이션 추가 호출
- 키워드·기관 화이트리스트를 JSON 설정 파일로 분리
- 입찰 마감 임박만 필터링, 또는 `bidNtceNm` 검색 전용 API 병행
- 제안서·실적표에 넣을 **한 페이지 요약**(기술스택·보안·운영) 템플릿을 `docs/` 에 유지
