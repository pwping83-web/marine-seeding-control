### 2-2. GPS 파싱 코드 완성 (readGpsFix 함수)

스케치의 `readGpsFix()` 함수는 기본값이 **고정 테스트 좌표**입니다. 반드시 실제 GPS UART 파싱으로 교체해야 합니다. GNSS 파싱에는 [TinyGPS++](https://github.com/mikalhart/TinyGPSPlus) 사용을 권장합니다.

```cpp
// ── 방법 1: TinyGPS++ 라이브러리 사용 (권장) ──────────────────
#include <TinyGPS++.h>
TinyGPSPlus gps;
HardwareSerial gpsSerial(1);

void setup() {
    // GPS UART: RX=GPIO16, TX=GPIO17 (보드에 따라 조정)
    gpsSerial.begin(9600, SERIAL_8N1, 16, 17);
}

static bool readGpsFix(double &lat, double &lng, float &speedKn, float &headingDeg) {
    unsigned long deadline = millis() + 200;  // 최대 200ms 대기
    while (millis() < deadline) {
        while (gpsSerial.available()) {
            gps.encode(gpsSerial.read());
        }
    }
    if (!gps.location.isValid()) return false;
    if (gps.location.age() > 5000) return false;  // 5초 이상 오래된 데이터 거부
    lat = gps.location.lat();
    lng = gps.location.lng();
    speedKn = gps.speed.knots();
    headingDeg = gps.course.isValid() ? gps.course.deg() : 0.0f;
    return true;
}

// ── 방법 2: 수동 NMEA 파싱 ($GPRMC) ────────────────────────────
// $GPRMC,HHMMSS.ss,A,DDMM.MMMM,N,DDDMM.MMMM,E,knots,hdg,date,...
static bool parseGprmc(const char *sentence, double &lat, double &lng,
                        float &speedKn, float &hdg) {
    // TinyGPS++ 사용이 훨씬 안정적이므로 방법 1 강력 권장
    return false;  // 직접 구현 필요
}
```

### 2-3. LTE 전용 보드 ([SIM7600G-H](https://www.simcom.com/product/SIM7600G-H.html) 등) 사용 시

기본 스케치는 WiFi + `WiFiClientSecure`를 사용합니다. LTE 모뎀으로 HTTP를 보내려면 모뎀 라이브러리([TinyGSM](https://github.com/vshymanskyy/TinyGSM) 등)로 HTTP 클라이언트([ArduinoHttpClient](https://github.com/arduino-libraries/ArduinoHttpClient) 등)를 교체해야 합니다.

```cpp
// TinyGSM + HTTPClient (LTE 전용)

#include <TinyGsmClient.h>
#include <ArduinoHttpClient.h>

TinyGsm modem(SerialAT);
TinyGsmClient gsmClient(modem);

// POST 방식만 다르고 URL·헤더·JSON 바디는 WiFi 방식과 동일
```

## 3. 현장 배선 절차

### 3-1. 배선 체크리스트 순서

```
□ 1. ESP32 단독 전원 테스트 (USB) → 시리얼 모니터 정상 출력 확인
□ 2. GPS 모듈 연결 → $GPRMC 문장 시리얼에 출력되는지 확인
□ 3. WiFi/LTE 연결 → "NTP synced" 출력 확인
□ 4. USE_ESP32_HTTP 0 상태에서 살포 핀(GPIO13) GND 단락
     → 시리얼에 {"records":[...]} JSON 출력 확인
□ 5. USE_ESP32_HTTP 1 변경 후 업로드 → curl 테스트 후 HTTP 200 확인
□ 6. 릴레이 모듈 연결 → GPIO11/12 HIGH/LOW 수동 확인 (멀티미터)
□ 7. 릴레이에 실제 부하(모터/밸브) 연결 → 동작 확인
□ 8. 방수 박스에 조립, 안테나 외부 마운트
□ 9. 부두 정박 상태에서 관제 웹에 마커 표시되는지 최종 확인
```

### 3-2. 투하 신호 배선 (GPIO 13)

```
방식 1 — 단순 테스트용 (택트 스위치):
  [스위치 한쪽] ──▶ ESP32 GPIO13
  [스위치 다른쪽] ──▶ ESP32 GND
  설정: DROP_TRIGGER_ON_HIGH = false

방식 2 — 투하장치 릴레이 출력 (건접점):
  [릴레이 NO 단자] ──▶ ESP32 GPIO13 (INPUT_PULLUP)
  [릴레이 COM 단자] ──▶ ESP32 GND
  설정: DROP_TRIGGER_ON_HIGH = false
  
방식 3 — PLC 24V 출력 (포토커플러 격리):
  입력 측 (24V 계통):
    [PLC 24V 출력] ──R470Ω──▶ [포토커플러 LED 애노드]
    [포토커플러 LED 카소드] ──▶ [PLC 출력 GND 또는 COM]
    (PLC 출력이 ON되면 LED가 켜지고 포토트랜지스터가 포화됨)

  출력 측 (3.3V 계통):
    [ESP32 3.3V] ──R10kΩ──┬──▶ [포토커플러 컬렉터]
                           └──▶ ESP32 GPIO13 (풀업 없어도 됨, 이미 외부 풀업)
    [포토커플러 이미터] ──▶ ESP32 GND

  동작: PLC 출력 ON → LED 켜짐 → 트랜지스터 포화 → 컬렉터~이미터 단락 → GPIO13 = LOW
  설정: DROP_TRIGGER_ON_HIGH = false  (하강 에지 = 투하)

  ※ 포토커플러 예시: PC817, TLP521, EL817 등 (NPN 출력형)
```

**제품 페이지(참고)**: [PC817 (onsemi)](https://www.onsemi.com/products/discrete-power-modules/optoelectronics/opto-isolators/pc817-series), [TLP521 (Toshiba)](https://www.toshiba.semicon-storage.com/ap-en/semiconductor/product/isolators-solid-state-relays/photocouplers/detail.TLP521-1.html).

## 4. 알려진 문제점 및 해결법

### 문제 1: GPS Cold Start 지연

**현상**: 전원 ON 후 첫 GPS 수신까지 수분 소요. 그동안 항적 전송 안 됨.  
**원인**: 위성 좌표 데이터(Almanac/Ephemeris) 재계산 필요.  
**해결**:
- GPS 모듈의 백업 배터리(CR1220 등) 장착 → Hot Start 가능 (5초 이내)
- 출항 전 항만에서 미리 전원 ON 후 수신 확인
- `readGpsFix()` 반환값이 `false`면 전송 건너뛰도록 코드에 이미 구현됨

### 문제 2: LTE 해상 끊김

**현상**: 연안에서 멀어질수록 LTE가 간헐적으로 끊김. `[vessel-track-ingest] HTTP fail` 로그.  
**원인**: 해상 LTE 커버리지 한계.  
**해결**:
- 재시도 로직 추가 (지수 백오프 방식)
- 끊긴 동안 데이터를 ESP32 `SPIFFS` 또는 `SD 카드`에 버퍼링 후 재전송

```cpp
// 재시도 예시 (스케치에 추가 가능)
bool postWithRetry(const String &body, int maxTries = 3) {
    for (int i = 0; i < maxTries; i++) {
        if (postVesselTrackJson(body)) return true;
        delay(2000 * (i + 1));  // 2초, 4초, 6초
        marineTaskWdtFeed();
    }
    return false;
}
```

### 문제 3: vessel_id 불일치로 지도 미표시

**현상**: 서버에는 데이터 쌓이는데 지도에 선박이 안 보임.  
**원인**: ESP32 `VESSEL_ID` vs 관제 웹 `VITE_VESSEL_LTE_ID` 불일치.  
**해결**:
```sql
-- Supabase SQL Editor에서 실제 저장된 ID 확인
SELECT DISTINCT vessel_id FROM vessel_track_points ORDER BY 1;
-- → 이 값을 .env의 VITE_VESSEL_LTE_ID에 정확히 복사·붙여넣기
```

### 문제 4: NTP 시간 미동기화로 recorded_at 오류

**현상**: `recorded_at`이 `0` 또는 부팅 후 ms값(작은 숫자)으로 저장됨.  
**원인**: `USE_ESP32_HTTP 0` 상태이거나, NTP 동기화 전에 POST.  
**해결**:
- `USE_ESP32_HTTP 1` 상태에서 `syncTimeNtp()` 이후에만 전송
- NTP 성공 여부를 체크하는 플래그 추가:

```cpp
static bool g_ntpOk = false;

void syncTimeNtp() {
    // UTC 기준(오프셋 0, DST 0)으로 설정 — KST로 하면 recorded_at이 9시간 오차!
    // unixEpochMsUtc()는 UTC ms를 반환하므로 반드시 UTC 기준으로 동기화해야 함
    configTime(0, 0, "pool.ntp.org", "time.google.com");
    for (int i = 0; i < 40; i++) {
        struct timeval tv;
        if (gettimeofday(&tv, nullptr) == 0 && tv.tv_sec > 1700000000) {
            g_ntpOk = true;
            return;
        }
        marineTaskWdtFeed();
        delay(500);
    }
    Serial.println(F("[ntp] 동기화 실패 — 전송 보류"));
}

// tickVesselTrack에서 NTP 확인
if (!g_ntpOk) return;  // NTP 안 됐으면 전송 건너뜀
```

> **핵심**: `configTime(9*3600, 0, ...)` KST로 설정하면 `gettimeofday()`가 UTC + 9시간 값을 반환하므로 `recorded_at`이 실제 시각보다 9시간 앞으로 기록됩니다. DB·관제 웹 모두 UTC를 기준으로 처리하므로 반드시 **`configTime(0, 0, ...)`** UTC로 설정하세요.

### 문제 5: 릴레이 모듈의 active-high vs active-low 혼동

**현상**: 모터가 반대로 동작하거나 항상 켜져 있음.  
**원인**: `RELAY_ACTIVE_LOW` 설정과 실제 모듈 사양 불일치.  
**해결**:
- 멀티미터로 릴레이 IN 핀이 LOW일 때 코일이 켜지는지 확인
- LOW가 ON이면: `RELAY_ACTIVE_LOW = true` (대부분의 2채널 릴레이 모듈)
- HIGH가 ON이면: `RELAY_ACTIVE_LOW = false`

### 문제 6: ESP32 전원 불안정 (LTE 전류 스파이크)

**현상**: LTE 모듈 송신 시 ESP32가 리셋되거나 WiFi/LTE 연결이 끊김.  
**원인**: LTE 모듈 송신 시 순간 전류가 USB 공급 한계(500mA) 초과.  
**해결**:
- USB 전원 대신 **배 전원 12V → DC-DC 벅컨버터 5V 3A 이상** 공급
- LTE 모듈 VCC 라인에 **1000μF 전해 캐패시터 병렬** 연결
- 접지 배선을 **별도로 굵게** (공통 GND 임피던스 최소화)

### 문제 7: 살포 오발 (전원 투입 시 진동)

**현상**: 전원 ON 시 GPIO13이 불안정하게 LOW 감지되어 살포 이벤트 오발.  
**원인**: 부팅 시 GPIO 리플.  
**해결**: `SEED_DROP_BOOT_GRACE_MS`(기본 3000ms)가 이미 방어함. 필요시 5000ms로 늘림.

### 문제 8: status 값 오류로 telemetry-ingest 400 반환

**현상**: `HTTP 400`, 서버 로그 `"invalid_record_shape"`.  
**원인**: `status` 필드에 "성공"/"실패"/"대기" 외의 값 (예: "OK", "success").  
**해결**: 코드에서 정확히 한국어 문자열 사용.
```cpp
rec["status"] = "성공";  // 이것만 유효
// rec["status"] = "OK";  // ❌ 400 오류
// rec["status"] = "success";  // ❌ 400 오류
```

### 문제 9: HTTPS TLS 핸드셰이크 실패 (CA 인증서)

**현상**: `postVesselTrackJson()` 항상 false 반환, 시리얼에 TLS 오류.  
**원인**: `client.setInsecure()` 미설정, 또는 CA 인증서 불일치.  
**해결**:
- 테스트/시험 중: `client.setInsecure()` 사용 (인증서 검증 생략)
- 운영 배포: Supabase의 루트 CA 인증서를 `client.setCACert(cert)` 로 설정

### 문제 10: 선박 아이콘 마커가 이동하지 않음 (항적 선만 늘어남)

**현상**: `vessel_track_points`에 점은 쌓이고 주황 항적 선은 늘어나는데, 지도의 선박 아이콘 마커 위치는 처음 그대로 고정.  
**원인**: `vessel-track-ingest` Edge Function은 `vessel_track_points`에만 삽입하고 `vessel_positions` 테이블을 갱신하지 않음. 선박 아이콘 마커는 `vessel_positions`를 기반으로 함.  
**해결 옵션**:

옵션 A — **브라우저에서 LTE 최신 점으로 자동 동기화** (관제 웹 코드 추가):
```typescript
// vessel_track_points 폴링 후 최신 점을 vessel_positions에 반영
const latestPt = trackPoints.at(-1);  // 가장 최신 점
if (latestPt) {
    await upsertVesselPosition(
        latestPt.vessel_id,
        VESSEL_USER_EMAIL,
        latestPt.lat,
        latestPt.lng,
        latestPt.heading_deg ?? 0,
        latestPt.speed_kn ?? 0
    );
}
```

옵션 B — **선박 담당 계정으로 관제 웹 브라우저를 선박에서도 열기** (가장 단순):  
배에 태블릿·스마트폰으로 관제 웹에 로그인하면 브라우저 GPS가 `vessel_positions`를 자동 업데이트.

### 문제 11: 해상 로밍 요금 폭탄

**현상**: LTE IoT 유심 데이터 요금이 예상보다 많이 청구됨.  
**원인**: 궤적 전송 주기가 너무 짧거나, 재시도가 과도함.  
**해결**:
- `POST_INTERVAL_MS`를 `120000`(2분) 이상으로 설정
- 한 번의 POST에 최대 100 포인트 배치 전송 활용
- LTE-M/NB-IoT 요금제 확인 (데이터량이 적어 저렴할 수 있음)

## 5. 현장 출항 전 최종 점검표

```
서버 점검:
□ curl로 vessel-track-ingest POST → HTTP 200 확인
□ curl로 telemetry-ingest POST → HTTP 200 확인
□ Supabase 대시보드에서 vessel_track_points에 행 1개 있는지 확인
□ Supabase 대시보드에서 seed_drop_records에 행 1개 있는지 확인

관제 웹 점검:
□ 브라우저에서 지도에 선박 마커 표시되는지 확인
□ 살포 마커 1개 표시되는지 확인
□ VITE_VESSEL_LTE_ID = ESP32 VESSEL_ID 일치 확인

ESP32 하드웨어 점검:
□ 시리얼 모니터에서 "NTP synced" 출력 확인
□ GPS lat/lng가 현재 위치와 대략 일치하는지 확인 (정박 위치)
□ 살포 핀(GPIO13) 단락 테스트 → {"records":[...]} POST 확인
□ 비상정지(GPIO9) 눌렀을 때 릴레이 OFF 확인
□ 시작/정지(GPIO10) 눌렀을 때 릴레이 ON/OFF 확인
□ 방수 박스 밀봉 확인, 케이블 글랜드 조임 확인
□ 안테나 고정 확인 (GPS, LTE 각각)
□ 배 전원 12V → DC-DC 5V 전압 확인 (멀티미터)
□ ENABLE_TASK_WDT 1 로 변경 후 최종 업로드 확인
```

## 6. 연동 구조 최종 요약

```
실제 배에서 관제 서버로 신호가 전달되는 전체 경로:

[GPS 위성]
  NMEA $GPRMC/$GPGGA 전파 신호
    ↓ (GPS 모듈 수신·파싱)
[GPS 모듈 UART TX → ESP32 UART RX]
  9600 baud, NMEA 텍스트
    ↓ (ESP32 TinyGPS++ 파싱 → lat/lng/speed/heading)
[ESP32 MCU]
  60초마다 JSON 직렬화
  {"vessel_id":"제3해양살포함","points":[{"lat":..,"lng":..,"recorded_at":..}]}
    ↓ (HTTPS POST, X-Device-Ingest-Key 헤더)
[LTE 모듈 → 기지국 → 인터넷]
    ↓
[Supabase Edge Function: vessel-track-ingest]
  인증 → 검증 → vessel_track_points INSERT
    ↓ (PostgreSQL 저장)
[관제 웹 12초 폴링 → Leaflet 지도 주황 선 업데이트]

[투하 신호 (건접점/PLC)]
  GPIO13 LOW 에지
    ↓ (50ms 디바운스 → 안전 로직 통과)
[ESP32]
  JSON: {"records":[{"id":..,"label":"A01","lat":..,"lng":..,"status":"성공",...}]}
    ↓ (HTTPS POST)
[Supabase Edge Function: telemetry-ingest]
  인증 → 검증 → seed_drop_records UPSERT
    ↓
[관제 웹 12초 폴링 → 지도에 살포 마커 + 이력 목록 추가]
```
