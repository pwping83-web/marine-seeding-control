# 05. 항해 경로 시작과 종료 — 출발·귀환 시 내비게이션 처리

> "배가 출발하고 돌아오면 내비게이션 경로를 끝내는지"

---

## 1. 항해 경로의 두 가지 개념

이 시스템에서 "항해 경로"는 두 가지 층위로 구분됩니다.

| 층위 | 이름 | 설명 | 관리 주체 |
|------|------|------|-----------|
| A | **사전 계획 경로** (WayPoint Plan) | 출항 전 관제 웹에서 입력한 작업 목표 경로 | 관제 웹 (TrackNavigationModal) |
| B | **실제 항적** (Track Points) | [ESP32](https://docs.espressif.com/projects/esp-idf/en/latest/esp32/hw-reference/esp32/get-started-devkitc.html)가 60초마다 전송한 실제 이동 경로 | ESP32 → vessel_track_points |

---

## 2. 사전 계획 경로 (WayPoint) 시작과 종료

### 2-1. 경로 입력 방법 (관제 웹)

관제 웹의 **TrackNavigationModal** 또는 **RouteNavPanel**에서:

1. 좌표 텍스트 입력 (lat, lng 형식, 한 줄에 한 점)
2. `parseLatLngTrackInput()` 함수로 파싱
3. 지도에 웨이포인트 마커로 표시

```
// 입력 형식 예시 (track-navigation.ts 기준)
34.712, 128.591
34.718, 128.595
34.725, 128.602
34.730, 128.610

// 또는 공백 구분
34.712 128.591
```

### 2-2. 경로 시작

- 관제 웹에서 **"항해 시작"** 버튼 클릭
- 웨이포인트 목록이 활성화되고 지도에 경로 선이 표시됨
- 현재 상태: **항해 중 (Active)**
- `localStorage` 또는 `sessionStorage`에 현재 활성 경로 저장 (브라우저 새로고침 시 유지)

### 2-3. 경로 종료 방법

경로를 종료하는 방법은 **세 가지**입니다.

| 종료 방식 | 트리거 | 동작 |
|----------|--------|------|
| **수동 종료** | 관제 웹에서 "항해 종료" 버튼 클릭 | 경로 비활성화, 항적 기록 보존 |
| **자동 귀환 판정** | 선박 위치가 출발점 반경 내로 복귀 | (별도 구현 필요 — 현재 자동화 없음) |
| **브라우저 세션 종료** | 탭 닫기 | 세션 기반 상태 초기화 |

> **중요**: 현재 시스템은 ESP32 단말이 귀항 여부를 감지하는 기능이 없습니다. 항해 경로 종료는 **관제 웹에서 수동으로** 처리합니다. 자동화가 필요하면 아래 §5 참고.

## 3. 실제 항적 (vessel_track_points) 관리

### 3-1. 항적 누적 시작

ESP32 전원 ON → NTP 시간 동기화 → 첫 번째 GPS Fix → 60초마다 자동 POST
→ `vessel_track_points`에 행 누적 시작

**항적이 자동으로 "시작"되는 이벤트는 별도로 없습니다.** 전원이 켜지고 GPS가 수신되는 순간부터 자동으로 쌓입니다.

### 3-2. 항적 누적 종료

ESP32 전원 OFF → 더 이상 POST 없음 → 마지막 점 이후 새 행 없음

**마찬가지로 명시적인 "종료" 이벤트 없이 전원이 꺼지면 자동으로 멈춥니다.**

### 3-3. 관제 웹에서의 오늘 항적 표시

관제 웹은 `vessel_track_points`에서 조건을 걸어 오늘 날짜의 항적만 표시합니다.

```sql
SELECT * FROM vessel_track_points
WHERE vessel_id = '제3해양살포함'
  AND recorded_at >= CURRENT_DATE::timestamptz
  AND recorded_at < (CURRENT_DATE + INTERVAL '1 day')::timestamptz
ORDER BY recorded_at ASC;
```

## 4. 출발~귀환 전체 시나리오

```
[출항 전 — 부두]
  관제 웹:
    1. 오늘 작업 계획 경로 입력 (WayPoint)
    2. 살포 구역 설정
    3. "항해 시작" 클릭 → 경로 활성화

  ESP32 점검:
    1. 전원 ON → 시리얼 모니터로 부팅 확인
    2. WiFi/LTE 연결 확인 → "NTP synced" 로그 확인
    3. GPS 수신 확인 → 첫 번째 POST 200 응답 확인
    4. 살포 핀 테스트 → GPIO13 수동 단락으로 살포 이벤트 1건 발생 확인

[항해 중]
  ESP32:
    - 60초마다 vessel-track-ingest POST
    - 살포 이벤트마다 telemetry-ingest POST
  
  관제 웹:
    - 12초마다 vessel_track_points 폴링 → 주황 항적 선 업데이트
    - 12초마다 seed_drop_records 폴링 → 살포 마커 추가

[귀항 — 부두 도착]
  ESP32:
    - 계속 POST 중 (전원이 켜져 있으면)
    - 또는 전원 OFF → 항적 자동 중단
  
  관제 웹:
    1. 오늘 항적·살포 이력 최종 확인
    2. "오늘 항적 보고서" 생성 (TodayTrackReportModal)
    3. "항해 종료" 클릭 → 사전 계획 경로 비활성화
    4. 필요시 데이터 내보내기

[다음 항해 준비]
  - vessel_track_points: 과거 항적은 그대로 DB에 남음 (삭제 불필요)
  - 관제 웹은 날짜 필터로 오늘/지난 항적 구분 조회 가능
```

## 5. 자동 귀항 감지 구현 방법 (고급 — 현재 미구현)

귀항 시 자동으로 경로를 종료하거나 알림을 보내려면 아래 방법을 추가로 구현해야 합니다.

### 방법 A: 관제 웹에서 지오펜스 감지

```typescript
// 출발점 좌표와 현재 선박 위치의 거리 계산
function haversineDistanceKm(
  lat1: number, lng1: number,
  lat2: number, lng2: number
): number {
  const R = 6371;
  const dLat = (lat2 - lat1) * Math.PI / 180;
  const dLng = (lng2 - lng1) * Math.PI / 180;
  const a = Math.sin(dLat/2)**2 +
            Math.cos(lat1 * Math.PI/180) *
            Math.cos(lat2 * Math.PI/180) *
            Math.sin(dLng/2)**2;
  return R * 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1-a));
}

// 선박이 귀항 반경(예: 200m) 내에 들어오면 자동 종료
const ARRIVAL_RADIUS_KM = 0.2;
const departurePoint = { lat: 34.720, lng: 128.581 };  // 출발 부두 좌표

if (haversineDistanceKm(
      currentVessel.lat, currentVessel.lng,
      departurePoint.lat, departurePoint.lng
    ) < ARRIVAL_RADIUS_KM) {
  // 항해 종료 처리
  endNavigation();
}
```

### 방법 B: ESP32에서 귀항 신호 전송

```cpp
// 배 전원 OFF 직전 귀항 완료 신호 전송
// (별도 "VOYAGE_END" 이벤트를 telemetry-ingest로 보내는 방식)
void sendVoyageEndEvent() {
    StaticJsonDocument<512> doc;
    JsonArray recs = doc.createNestedArray("records");
    JsonObject r = recs.createNestedObject();
    r["id"] = String("voyage-end-") + String(unixEpochMsUtc());
    r["label"] = "VOYAGE_END";        // 관제 웹에서 이 라벨을 감지해 항해 종료 처리
    r["drop_time"] = currentTimeStr();
    r["lat"] = lastLat;
    r["lng"] = lastLng;
    // "대기"로 설정 — 살포 이벤트가 아님을 구분
    // (서버는 "성공"|"실패"|"대기" 세 값만 허용)
    r["status"] = "대기";
    r["recorded_at"] = unixEpochMsUtc();
    // ... POST
}
```

> **주의**: `VOYAGE_END` 레코드가 `seed_drop_records`에 들어가면 살포 건수 통계에 포함될 수 있습니다. 관제 웹에서 `label == "VOYAGE_END"` 조건으로 살포 통계에서 제외하는 필터를 추가해야 합니다.

## 6. "신선도" 타임아웃 — LTE 신호 끊김 감지

관제 웹은 `vessel_track_points`의 최신 점을 기준으로 경과 시간을 확인합니다.

| 경과 시간 | 관제 웹 표시 |
|----------|-------------|
| 0~25분 | 정상 (녹색 또는 무표시) |
| 25분 초과 | "신호 없음" 경고 표시 |

이 기준값은 코드에서 변경 가능합니다. 항해 중 LTE 통신 불가 구역(원양 등)에서는 경고가 뜰 수 있으므로 운영 해역에 맞게 조정하세요.
