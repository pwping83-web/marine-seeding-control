## 1. MCU (메인 컨트롤러)

### 1-1. 요구 사양

| 항목 | 최소 요구 | 권장 |
|------|-----------|------|
| 아키텍처 | 32비트 ARM/Xtensa | ESP32 (Xtensa LX6 듀얼코어) |
| RAM | ≥ 256 KB | ≥ 320 KB (ESP32 WROOM 기준 520 KB SRAM) |
| 플래시 | ≥ 4 MB | 4~8 MB (TLS 스택 + ArduinoJson 여유) |
| UART 포트 | ≥ 2개 | 3개 이상 (GPS·LTE·디버그 시리얼 분리) |
| GPIO | ≥ 10핀 | 30핀 이상 (입력·출력·비상·릴레이 분리) |
| HTTPS 지원 | 필수 | `WiFiClientSecure` 또는 TinyGSM + TLS |
| 동작 전압 | 3.3 V 로직 | 3.3 V (5 V TTL LTE 모듈이면 레벨 시프터 필요) |
| 동작 온도 | -10 ~ +70 °C | -20 ~ +85 °C (해상 동절기·하절기 대응) |

| **현장 권장 (일체형)** | **[LilyGO T-SIM7600G-H](https://github.com/Xinyuan-LilyGO/T-SIM7600X)**, [LilyGO T-A7670](https://github.com/Xinyuan-LilyGO/LilyGo-T-A7670), [TTGO T-Call](https://github.com/Xinyuan-LilyGO/LilyGo-T-Call-SIM800) | ESP32 + SIM 슬롯 + GNSS 단자 한 장. 구매 전 밴드 확인 필수 |
| 산업용 | 방수 RTU 단말기 (박스형) | IP65~IP67, 차량·선박 전원, 인증 제품 |

### 1-3. 핀 배정

| GPIO | 방향 | 용도 | 비고 |
|------|------|------|------|
| **13** | INPUT_PULLUP | 살포 신호 수신 핀 | PLC·센서·수동 버튼 펄스 입력 |
| GPS TX → ESP32 RX | UART RX | GPS 모듈 NMEA 데이터 수신 | 예: GPIO16 (HardwareSerial 1) |
| LTE TX/RX | UART2 | LTE 모뎀 AT 명령 | TinyGSM 사용 시 |

**▶ 스케치 B: `esp32_marine_telemetry_motor_relay.ino` — 텔레메트리 + 모터·릴레이 제어 추가**

| GPIO | 방향 | 용도 | 비고 |
|------|------|------|------|
| **9** | INPUT_PULLUP | 비상정지 버튼 | LOW = 비상 (EMERGENCY_STOP_ACTIVE_LOW = true) |
| **10** | INPUT_PULLUP | 모터 시작/정지 토글 버튼 | 눌렸을 때 LOW |
| **11** | OUTPUT | 릴레이 A (투하 모터 정방향·밸브 A) | RELAY_ACTIVE_LOW = true → LOW가 ON |
| **12** | OUTPUT | 릴레이 B (투하 모터 역방향·밸브 B) | 동일 |
| **13** | INPUT_PULLUP | 살포 신호 수신 핀 | PLC·센서 펄스 입력 |
| GPS TX → ESP32 RX | UART RX | GPS 모듈 NMEA 데이터 수신 | 예: GPIO16 |
| LTE TX/RX | UART2 | LTE 모뎀 AT 명령 | TinyGSM 사용 시 |

> **주의**: GPIO9, 10, 11, 12는 스케치 B(모터 릴레이)에서만 사용. 일부 ESP32 보드에서 **GPIO9는 내부 플래시와 공유**될 수 있어 문제 발생 시 다른 핀으로 변경 필요.

## 2. GPS 모듈 — 사양 및 프로토콜

### 2-1. 표준 NMEA-0183 프로토콜

GPS 모듈은 **NMEA-0183** 규격 텍스트 문장을 UART로 출력합니다.

**주요 문장 형식**

```
$GPRMC,HHMMSS.ss,A,LLLL.LL,a,YYYYY.YY,a,x.x,x.x,DDMMYY,x.x,a*hh<CR><LF>
```

| 필드 | 내용 | 예시 |
|------|------|------|
| `$GPRMC` | 문장 식별자 (권장 최소 문장) | `$GPRMC` |
| HHMMSS.ss | UTC 시각 (시분초.소수) | `023451.00` |
| A | 상태 (A=유효, V=무효) | `A` |
| LLLL.LL | 위도 (DDMM.MMMMM) | `3449.2010` |
| a | N/S | `N` |
| YYYYY.YY | 경도 (DDDMM.MMMMM) | `12831.2340` |
| a | E/W | `E` |
| x.x | 대지 속도 (노트) | `4.20` |
| x.x | 진행 방위 (도) | `045.0` |
| DDMMYY | 날짜 | `140526` |

**$GPGGA 문장 (위성 수·고도 포함)**

```
$GPGGA,HHMMSS.ss,LLLL.LL,a,YYYYY.YY,a,q,nn,x.x,x.x,M,x.x,M,x.x,xxxx*hh
```

| 필드 | 내용 |
|------|------|
| q | GPS 품질 (0=미수신, 1=GPS, 2=DGPS/RTK Float, 4=RTK Fix) |
| nn | 사용 위성 수 |
| x.x (4번째) | HDOP (수평 정밀도 저하율, 낮을수록 좋음) |
| x.x (5번째) | 타원체 고도 (m) |

### 2-2. 파싱 코드 (ESP32 Arduino)

```cpp
// TinyGPS++ 라이브러리 사용 권장
#include <TinyGPS++.h>
TinyGPSPlus gps;
HardwareSerial gpsSerial(1);  // UART1

void setup() {
    gpsSerial.begin(9600, SERIAL_8N1, 16, 17);  // RX=16, TX=17
}

bool readGpsFix(double &lat, double &lng, float &speedKn, float &headingDeg) {
    while (gpsSerial.available()) {
        gps.encode(gpsSerial.read());
    }
    if (!gps.location.isValid() || !gps.location.isUpdated()) return false;
    lat = gps.location.lat();
    lng = gps.location.lng();
    speedKn = gps.speed.knots();
    headingDeg = gps.course.deg();
    return true;
}
```

## 3. RTK 보정 프로토콜

### 3-1. RTK 개요

RTK(Real-Time Kinematic)는 기준국(Base Station)의 보정 데이터를 이용하여 GPS 오차를 **cm급**으로 줄이는 기술입니다.

| 방식 | 정확도 | 조건 |
|------|--------|------|
| 일반 GPS | 2~10m | 위성 수신만으로 |
| SBAS/DGNSS | 1~3m | 국토지리원 DGPS 기준국 |
| RTK Float | 20~50cm | 기준국 통신 연결, 위성 부족 |
| **RTK Fix** | **1~3cm** | 기준국 연결 + 충분한 위성 |

### 3-2. RTCM 3.x 프로토콜

RTK 보정 데이터는 **RTCM SC-104 버전 3.x** 바이너리 형식으로 전송됩니다.

**주요 메시지 타입**

| 메시지 번호 | 내용 |
|-------------|------|
| 1001~1004 | GPS RTK 관측 데이터 |
| 1005 | 기준국 좌표 (정적) |
| 1006 | 기준국 좌표 + 안테나 높이 |
| 1009~1012 | GLONASS RTK 관측 데이터 |
| 1033 | 수신기·안테나 정보 |
| 1074~1077 | MSM GPS 관측 데이터 (고정밀) |

### 3-3. NTRIP 프로토콜 (네트워크 RTK)

해상에서는 **NTRIP(Networked Transport of RTCM via Internet Protocol)**으로 LTE를 통해 기준국 보정 데이터를 수신합니다.

```
연결 구조:
  ESP32 (NTRIP Client) ─── LTE ───▶ NTRIP Caster (서버) ◀─── NTRIP Server (기준국)

주소 예시:
  국토지리원 VRS: vrs.ngii.go.kr:2101
  마운트포인트:   RTCM32_MSM (전국 가상 기준점)
  포트: 2101
  프로토콜: HTTP 기반 NTRIP 1.0 / 2.0
```

**NTRIP 연결 헤더 예시**

```
GET /RTCM32_MSM HTTP/1.1
Host: vrs.ngii.go.kr:2101
Ntrip-Version: Ntrip/2.0
Authorization: Basic BASE64(ID:PASSWORD)
User-Agent: NTRIP ESP32Client/1.0
Connection: close
```

### 3-4. RTK 수신 코드 구조 (ESP32)

```cpp
// RTCM 보정 데이터를 GPS 모듈의 UART에 직접 전달
WiFiClient ntripClient;
HardwareSerial gpsSerial(1);

void applyRtcmCorrection() {
    // NTRIP Caster에서 읽은 RTCM 바이트를 GPS 모듈에 전달
    while (ntripClient.available()) {
        uint8_t buf[256];
        int len = ntripClient.read(buf, sizeof(buf));
        if (len > 0) {
            gpsSerial.write(buf, len);  // GPS 모듈에 RTCM 주입
        }
    }
}
```

### 3-5. RTK 적용 시 $GPGGA 품질 필드 변화

| q 값 | 의미 | 정확도 |
|------|------|--------|
| 0 | 미수신 (전송 차단 권장) | - |
| 1 | 일반 GPS | ~5m |
| 2 | DGPS / RTK Float | ~0.5m |
| **4** | **RTK Fix (최고)** | **~1~3cm** |

> **실무 주의**: 해상에서는 파도·선체 진동으로 RTK Fix가 계속 유지되지 않을 수 있습니다. q=1 기준으로도 살포 기록은 허용하되, q=0이면 전송 차단하도록 설계합니다.

## 4. 투하장치 연동 프로토콜

### 4-1. 신호 인터페이스 방식

투하장치(Dropper)는 MCU와 다음 방식으로 연결합니다.

**방식 A: 건접점(Dry Contact) — 가장 안전**

```
투하장치 제어 출력
  ├── NO (Normally Open) 단자 ──▶ ESP32 GPIO13 (INPUT_PULLUP)
  └── COM 단자              ──▶ ESP32 GND

동작: 투하 순간 릴레이 닫힘(NO→COM 연결) → GPIO13 = LOW
설정: DROP_TRIGGER_ON_HIGH = false (하강 에지 = 투하)
```

**방식 B: 오픈 드레인 출력 (PLC 등)**

```
PLC 디지털 출력(트랜지스터 컬렉터 오픈 드레인)
  └── 출력 단자 ──▶ ESP32 GPIO13 (INPUT_PULLUP)
      (투하 시 PLC가 내부적으로 GND로 당김 → LOW)
```

**방식 C: 5V/24V 논리 레벨 (레벨 변환 필요)**

```
PLC 24V 출력 ──▶ 포토커플러(옵토아이솔레이터) ──▶ ESP32 GPIO13
              또는
PLC 5V 출력  ──▶ 레벨 시프터 (5V→3.3V) ──▶ ESP32 GPIO13
```

### 4-2. PLC 연동 I/O 신호 명세

| 신호명 | 방향 | 타입 | 핀 | 설명 |
|--------|------|------|-----|------|
| SEED_DROP_SIGNAL | PLC→MCU | 디지털 입력 | GPIO 13 | 투하 1건 발생 펄스 |
| MOTOR_START | MCU→PLC | 디지털 출력 | GPIO 11 (릴레이 A) | 투하 모터/밸브 A 구동 |
| MOTOR_STOP | MCU→PLC | 디지털 출력 | GPIO 12 (릴레이 B) | 투하 모터/밸브 B 구동 |
| EMERGENCY_STOP | 현장→MCU | 디지털 입력 | GPIO 9 | 비상정지 (LOW=비상) |
| RUN_STOP_TOGGLE | 현장→MCU | 디지털 입력 | GPIO 10 | 모터 시작/정지 토글 |

### 4-3. 릴레이 모듈 배선

```
ESP32 GPIO 11 ──▶ 릴레이 모듈 IN1  (active-low: LOW = 코일 ON)
ESP32 GPIO 12 ──▶ 릴레이 모듈 IN2
ESP32 3.3V    ──▶ 릴레이 모듈 VCC (또는 5V 별도 공급)
ESP32 GND     ──▶ 릴레이 모듈 GND

릴레이 모듈 COM  ──▶ 모터/밸브 전원
릴레이 모듈 NO   ──▶ 모터/밸브 부하

※ 중요: 모터(유도부하) 사용 시 릴레이 모듈에
         내장 플라이백 다이오드가 있는 제품 사용.
         없으면 외부에 다이오드(1N4007 등) 추가 필수.
```

### 4-4. 투하 타이밍 다이어그램

`DROP_TRIGGER_ON_HIGH` 설정에 따라 트리거 에지 방향이 다릅니다.

**DROP_TRIGGER_ON_HIGH = true (기본값, 스케치 기본 설정)**

```
GPIO13 상태:
  HIGH                  ┌─────────┐
  LOW  ────────────────┘         └────────────
       (평소 LOW)      ↑   ↑
                 변화감지 50ms 후 에지 처리

투하 판정: 상승 에지 (LOW → HIGH) 감지 후 50ms 안정 확인
→ sendDropTelemetryNow() 호출 → HTTPS POST
→ 50ms 이후 다시 LOW로 내려와도 이미 전송 완료
```

**DROP_TRIGGER_ON_HIGH = false (건접점·수동 스위치 방식)**

```
GPIO13 상태:
  HIGH ────────────────┐         ┌────────────
  LOW                  └─────────┘
       (풀업으로 평소 HIGH)↑   ↑
                    변화감지 50ms 후 에지 처리

투하 판정: 하강 에지 (HIGH → LOW) 감지 후 50ms 안정 확인
→ sendDropTelemetryNow() 호출
```

```
최소 간격 (SEED_DROP_MIN_INTERVAL_MS = 800ms):
  연속 투하 시 800ms 이내 두 번째 신호는 무시
```
