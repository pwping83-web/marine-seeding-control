# 02. 신호 프로토콜 명세 — 실제 코딩값 (JSON · HTTP)

> **"어떤 신호(코딩값)를 보냈을 때 우리 서버에 배 위치가 잡히고 투하가 기록되는가"**  
> 이 문서는 그 실제 HTTP 요청 형식을 정확히 기술합니다.

---

## 1. 공통 HTTP 헤더

두 API 모두 아래 헤더를 포함해야 합니다.

```http
POST https://{SUPABASE_PROJECT_REF}.supabase.co/functions/v1/{endpoint}
Content-Type: application/json
X-Device-Ingest-Key: {DEVICE_INGEST_SECRET}
```

| 헤더 | 값 | 설명 |
|------|-----|------|
| `Content-Type` | `application/json` | JSON 바디 전송 |
| `X-Device-Ingest-Key` | Supabase Secret 설정값 | 이 값이 서버의 `DEVICE_INGEST_SECRET`과 일치해야만 수신 |

> **중요**: `X-Device-Ingest-Key`가 틀리면 HTTP 401 반환. 배가 출항하기 전 반드시 확인.

## 2. 선박 위치 전송 API — `vessel-track-ingest`

### 2-1. 엔드포인트

```
POST https://{REF}.supabase.co/functions/v1/vessel-track-ingest
```

### 2-2. 요청 JSON 형식

```json
{
  "vessel_id": "제3해양살포함",
  "points": [
    {
      "lat": 34.8200,
      "lng": 128.5200,
      "recorded_at": 1747177200000,
      "speed_kn": 4.2,
      "heading_deg": 45.0
    }
  ]
}
```

### 2-3. 필드 상세

| 필드 | 타입 | 필수 | 범위 | 설명 |
|------|------|------|------|------|
| `vessel_id` | string | ✅ | 1~120자 | 선박 고유 ID. 관제 웹 `VITE_VESSEL_LTE_ID`와 **동일해야** 지도에 표시됨 |
| `points` | array | ✅ | 1~100개 | 위치 포인트 배열 (한 번에 최대 100개) |
| `points[].lat` | number | ✅ | -90 ~ 90 | 위도 (WGS84 도 단위) |
| `points[].lng` | number | ✅ | -180 ~ 180 | 경도 (WGS84 도 단위) |
| `points[].recorded_at` | number | ✅ | Unix ms | GPS 측정 시각 (밀리초). NTP 동기화된 `gettimeofday()` 값 사용 권장 |
| `points[].speed_kn` | number | 선택 | 0 ~ 120 | 대지 속도 (노트). GPS $GPRMC 속도 필드 |
| `points[].heading_deg` | number | 선택 | 0 ~ 360 | 진행 방위 (도). GPS $GPRMC 방위 필드 |

### 2-4. 응답

```json
// 성공 (HTTP 200)
{ "ok": true, "accepted": 1 }

// 실패 예시
{ "error": "invalid_device_key" }          // 401: Secret 불일치
{ "error": "invalid_vessel_id" }           // 400: vessel_id 없거나 너무 긺
{ "error": "invalid_point_shape" }         // 400: 좌표 범위 이탈 또는 필드 누락
{ "error": "rate_limited_per_ip" }         // 429: IP당 초당 15건 초과
{ "error": "payload_too_large" }           // 413: 64KB 초과
```

### 2-5. ESP32 실제 전송 코드

```cpp
// 선박 궤적 JSON 빌드 및 전송
StaticJsonDocument<512> doc;
doc["vessel_id"] = "제3해양살포함";  // VESSEL_ID 변수값
JsonArray pts = doc.createNestedArray("points");
JsonObject p = pts.createNestedObject();

p["lat"] = 34.8200;           // readGpsFix()에서 읽은 실제 위도
p["lng"] = 128.5200;          // readGpsFix()에서 읽은 실제 경도
p["recorded_at"] = unixEpochMsUtc();  // NTP 동기화 후 gettimeofday()
p["speed_kn"] = 4.2;          // GPS $GPRMC 속도
p["heading_deg"] = 45.0;      // GPS $GPRMC 방위

String body;
serializeJson(doc, body);
// → {"vessel_id":"제3해양살포함","points":[{"lat":34.82,"lng":128.52,"recorded_at":1747177200000,"speed_kn":4.2,"heading_deg":45.0}]}

// HTTPS POST
WiFiClientSecure client;
client.setInsecure();  // 운영 시 CA 인증서 검증 권장
HTTPClient http;
http.begin(client, "https://xxxx.supabase.co/functions/v1/vessel-track-ingest");
http.setTimeout(20000);
http.addHeader("Content-Type", "application/json");
http.addHeader("X-Device-Ingest-Key", "your-secret-here");
int code = http.POST(body);
// code == 200: 성공
```

## 3. 살포 기록 전송 API — `telemetry-ingest`

### 3-1. 엔드포인트

```
POST https://{REF}.supabase.co/functions/v1/telemetry-ingest
```

### 3-2. 요청 JSON 형식

```json
{
  "records": [
    {
      "id": "ship-제3해양살포함-1747177200000",
      "label": "A01",
      "drop_time": "14:30:45",
      "lat": 34.8215,
      "lng": 128.5235,
      "status": "성공",
      "recorded_at": 1747177200000,
      "verification_mismatch": false
    }
  ]
}
```

### 3-3. 필드 상세

| 필드 | 타입 | 필수 | 허용값 | 설명 |
|------|------|------|--------|------|
| `records` | array | ✅ | 1~80개 | 살포 기록 배열 |
| `records[].id` | string | ✅ | 1~200자 | 살포 이벤트 고유 ID. **중복 전송 방지**용 (upsert). 보통 `선박ID + 밀리초` 조합 |
| `records[].label` | string | ✅ | ~500자 | 살포 위치·구역명. 예: `"A01"`, `"구역3-남쪽"` |
| `records[].drop_time` | string | ✅ | `"HH:MM:SS"` | 살포 발생 시각 문자열. GPS RTC 또는 NTP 시각 사용 |
| `records[].lat` | number | ✅ | -90 ~ 90 | 살포 발생 위치 위도 |
| `records[].lng` | number | ✅ | -180 ~ 180 | 살포 발생 위치 경도 |
| `records[].status` | string | ✅ | `"성공"`, `"실패"`, `"대기"` | **이 3개 값만 허용**. 다른 값이면 400 오류 |
| `records[].recorded_at` | number | ✅ | Unix ms | 기록 시각 (밀리초) |
| `records[].verification_mismatch` | boolean | 선택 | true/false | GPS 좌표 이상 시 true 표기 (검증 불일치) |

### 3-4. 응답

```json
// 성공 (HTTP 200)
{ "ok": true, "accepted": 1 }

// 실패 예시
{ "error": "invalid_device_key" }          // 401
{ "error": "invalid_records_array" }       // 400: records 없거나 80개 초과
{ "error": "invalid_record_shape" }        // 400: status가 "성공"/"실패"/"대기" 외 값
```

### 3-5. ESP32 실제 전송 코드

```cpp
// 살포 이벤트 JSON 빌드 및 전송
StaticJsonDocument<768> doc;
JsonArray recs = doc.createNestedArray("records");
JsonObject r = recs.createNestedObject();

// 고유 ID: 중복 방지 (같은 ID로 다시 보내면 upsert로 덮어씀)
String idStr = String("ship-제3해양살포함-") + String(unixEpochMsUtc());
r["id"] = idStr.c_str();

r["label"] = "A01";                    // 살포 구역 또는 라벨
r["drop_time"] = "14:30:45";           // GPS RTC 시각 (HH:MM:SS)
r["lat"] = 34.8215;                    // GPS 위도
r["lng"] = 128.5235;                   // GPS 경도
r["status"] = "성공";                  // "성공" | "실패" | "대기"
r["recorded_at"] = unixEpochMsUtc();   // Unix 밀리초
r["verification_mismatch"] = false;

String body;
serializeJson(doc, body);
// → {"records":[{"id":"ship-제3해양살포함-1747177200000","label":"A01","drop_time":"14:30:45","lat":34.8215,"lng":128.5235,"status":"성공","recorded_at":1747177200000,"verification_mismatch":false}]}
```

## 4. DB 테이블 구조 (서버 측)

### 4-1. vessel_track_points (선박 항적)

```sql
CREATE TABLE vessel_track_points (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  vessel_id   TEXT NOT NULL,          -- "제3해양살포함"
  recorded_at TIMESTAMPTZ NOT NULL,   -- GPS 시각
  lat         DOUBLE PRECISION NOT NULL,
  lng         DOUBLE PRECISION NOT NULL,
  speed_kn    DOUBLE PRECISION,       -- 노트
  heading_deg DOUBLE PRECISION,       -- 도
  source      TEXT DEFAULT 'arduino-lte',
  created_at  TIMESTAMPTZ DEFAULT now()
);
```

### 4-2. vessel_positions (현재 위치 — 실시간)

```sql
CREATE TABLE vessel_positions (
  vessel_id    TEXT PRIMARY KEY,
  user_email   TEXT NOT NULL,
  lat          DOUBLE PRECISION NOT NULL,
  lng          DOUBLE PRECISION NOT NULL,
  heading      DOUBLE PRECISION DEFAULT 0,
  speed_knots  DOUBLE PRECISION DEFAULT 0,
  updated_at   TIMESTAMPTZ DEFAULT NOW()
);
```

> **vessel_positions vs vessel_track_points 차이**  
> - `vessel_positions`: **현재 위치만** (1척 = 1행, upsert로 덮어씀). 지도의 실시간 마커용.  
> - `vessel_track_points`: **모든 이력 누적** (1척 = 수백~수만 행). 항적 선 그리기용.

### 4-3. seed_drop_records (살포 기록)

```sql
CREATE TABLE seed_drop_records (
  id                    TEXT PRIMARY KEY,  -- 장치가 보낸 고유 ID
  label                 TEXT,
  drop_time             TEXT,              -- "HH:MM:SS" 형식
  lat                   DOUBLE PRECISION,
  lng                   DOUBLE PRECISION,
  status                TEXT,              -- "성공"|"실패"|"대기"
  recorded_at           BIGINT,            -- Unix ms
  verification_mismatch BOOLEAN DEFAULT false
);
```

## 5. 전송 주기 설정값 요약

| 항목 | 코드 상수 | 기본값 | 변경 가능 범위 | 위치 |
|------|-----------|--------|---------------|------|
| 선박 궤적 전송 주기 | `POST_INTERVAL_MS` | 60,000 ms (1분) | 10초~10분 | [ESP32](https://docs.espressif.com/projects/esp-idf/en/latest/esp32/hw-reference/esp32/get-started-devkitc.html) 스케치 |
| 살포 신호 디바운스 | `DROP_DEBOUNCE_MS` | 50 ms | 20~200 ms | [ESP32](https://docs.espressif.com/projects/esp-idf/en/latest/esp32/hw-reference/esp32/get-started-devkitc.html) 스케치 |
| 살포 최소 간격 | `SEED_DROP_MIN_INTERVAL_MS` | 800 ms | 500ms~5초 | [ESP32](https://docs.espressif.com/projects/esp-idf/en/latest/esp32/hw-reference/esp32/get-started-devkitc.html) 스케치 |
| 60초당 최대 살포 시도 | `SEED_DROP_MAX_ATTEMPTS_PER_60S` | 24회 | 1~60회 | [ESP32](https://docs.espressif.com/projects/esp-idf/en/latest/esp32/hw-reference/esp32/get-started-devkitc.html) 스케치 |
| 부팅 유예 시간 | `SEED_DROP_BOOT_GRACE_MS` | 3,000 ms | 1~10초 | [ESP32](https://docs.espressif.com/projects/esp-idf/en/latest/esp32/hw-reference/esp32/get-started-devkitc.html) 스케치 |
| 웹 궤적 폴링 주기 | `VITE_VESSEL_LTE_POLL_MS` | ~12,000 ms | 5초~120초 | .env |
| 웹 살포 폴링 주기 | `VITE_SEED_DROP_POLL_MS` | ~12,000 ms | 4초~120초 | .env |

## 6. 레이트 리밋 (서버 제한)

| API | IP당 한도 | 글로벌 한도 | 단일 요청 최대 |
|-----|-----------|-------------|----------------|
| vessel-track-ingest | 15건/초 | 4,000건/초 | 100 포인트 |
| telemetry-ingest | 25건/초 | 8,000건/초 | 80 레코드 |

> 제한 초과 시 HTTP 429 반환. ESP32는 `code == 200`이 아니면 로그 출력 (재시도 전략은 현장 필요에 따라 추가).

## 7. curl 테스트 명령 (PC에서 서버 확인)

### 선박 위치 전송 테스트
```bash
curl -X POST https://YOUR_REF.supabase.co/functions/v1/vessel-track-ingest \
  -H "Content-Type: application/json" \
  -H "X-Device-Ingest-Key: YOUR_SECRET" \
  -d '{
    "vessel_id": "제3해양살포함",
    "points": [
      {"lat": 34.82, "lng": 128.52, "recorded_at": 1747177200000, "speed_kn": 4.2, "heading_deg": 45.0}
    ]
  }'
# 기대 응답: {"ok":true,"accepted":1}
```

### 살포 기록 전송 테스트
```bash
curl -X POST https://YOUR_REF.supabase.co/functions/v1/telemetry-ingest \
  -H "Content-Type: application/json" \
  -H "X-Device-Ingest-Key: YOUR_SECRET" \
  -d '{
    "records": [
      {
        "id": "test-001",
        "label": "A01",
        "drop_time": "14:30:45",
        "lat": 34.82,
        "lng": 128.52,
        "status": "성공",
        "recorded_at": 1747177200000,
        "verification_mismatch": false
      }
    ]
  }'
# 기대 응답: {"ok":true,"accepted":1}
```

> PC에서 curl로 먼저 200 응답을 확인한 후 ESP32로 넘어가는 것이 디버깅에 효율적입니다.
