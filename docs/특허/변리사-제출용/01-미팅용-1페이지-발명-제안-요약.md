# 변리사 미팅용 1페이지 발명 제안 요약

> 본 요약은 저장소 구현(`src/`, `supabase/functions/`, `arduino/` 등)과 **맞추어** 기술 서술을 정리한 것입니다. 등록 가능성·청구항·출원 전략은 변리사 판단 및 선행기술 조사가 필요합니다.

---

## 1. 발명의 명칭 (가칭)

**해양 기상 데이터 연동, 살포 계획(구역 점수·작업 한도) 및 장애물 회피 항로 생성을 포함한 종자 살포 관제 시스템**

---

## 2. 배경 기술 및 과제 (3줄 요약)

1. 해양 종자 살포는 수동·경험 의존이 커 투하 지점·작업 강도의 규칙성이 떨어지고, 기상 악화에 따른 작업 조정을 데이터로 일관되게 남기기 어렵다.  
2. 해상 LTE 단절이 잦고, 다수 선박에서 DB·지도 API 키를 단말에 두면 보안 위험이 커지며, 살포 순간 트래픽과 주기 궤적이 한 통로에 몰리면 수집 서버 운영이 어려워진다.  
3. 본 발명은 **기상·해역 조건을 반영한 계획 모듈**, **장애물 회피 항로 및 웨이포인트화**, **엣지 게이트웨이 이원 수집**, **살포 점 기반 면적 근사·이력 시각화**로 상기 과제를 완화하는 것을 목표로 한다.

---

## 3. 발명의 구체적 해결 수단 (핵심 기술 표)

| 기술 포인트 (보호 후보) | 세부 구현 및 차별성 (실시예·코드 근거) |
|-------------------------|----------------------------------------|
| **포인트 1** 기상·해역 조건 연동 살포 계획 및 장애물 회피 항로·웨이포인트 | 기상청 Open API 등(`kma-weather.ts`)으로 풍속·파고·시정 등을 반영하고, 해역 격자에 대해 수심·수온·해류 **시뮬** 및 기상 패널티를 포함한 **구역 점수화**·**일일 살포 한도**를 산출한다(`seeding-plan-ai.ts`). **A\***는 암초·보호구역 등 **장애물 반경 내 격자를 탐색 제외**하고, 각 스텝 비용은 **haversine 거리(km)**에 **풍향·풍속·돌풍에 따른 기상 배수(`weatherStepCostMultiplier`)**를 곱해 누적한다(역풍 성분이 클수록 비용 증가, 상한 있음). 기상 객체를 넘기지 않으면 배수 1로 **거리만** 사용한다. 항로는 역추적 후 **경로 간소화(예: 일정 간격 샘플)**로 웨이포인트 시퀀스를 만든다. 기상에 따른 **PID 파라미터 제안·Arduino 스케치 스니펫**을 병행한다(`suggestPidParams`, UI `SeedingPlanPanel` 등). |
| **포인트 2** 살포 이벤트와 선박 궤적의 **이원화 엣지 수집** | 단말은 **HTTPS POST**로만 Supabase Edge에 접속하고, 헤더 **`X-Device-Ingest-Key`**(서버 시크릿과 일치)로 검증한다. **`telemetry-ingest`**(살포 `records` 등)와 **`vessel-track-ingest`**(궤적 `points` 등)를 **별도 함수**로 두고, **본문 최대 바이트·배열 최대 길이·IP당·글로벌 초당 요청 상한**을 서로 다르게 둔다(예: 살포 쪽 IP 25/s·글로벌 8000/s, 궤적 쪽 IP 15/s·글로벌 4000/s — 배포 시 코드 상수 기준). 좌표·`recorded_at` 범위 등 **스키마 검증** 후 DB upsert. **망 단절 시 단말 로컬 큐·재전송·중복 제거**는 문서상 권장 설계(`arduino/아두이노-스케치-모음/문서/02-해상-통신-범위-한계.md` 등)이며, **펌웨어 버전별 실시예는 출원 시 명세와 일치시킬 것.** |
| **포인트 3** 살포 좌표 기반 **유효 면적(ha) 근사** | 서버·클라이언트에서 살포 점 집합에 **Graham scan 계열 Convex Hull**을 적용한 뒤, 위경도를 km로 근사 투영하여 **Shoelace(가우스) 면적**으로 ha를 산출한다(`seeding-day-eval.ts`). 점 **3개 미만**은 0 처리. **격자 정밀 측량이 아닌 관제·보고 참고치**임을 전제한다. 물리 스위치 **디바운스(ms) 조정**은 예시 스케치에 주석으로 명시(`esp32_marine_telemetry_motor_relay.ino` 등). **`verification_mismatch`** 등 검수 플래그와 UI 색 구분은 데이터 품질 표시에 활용된다. |
| **포인트 4** 기록 시각 기반 **연령(Age) 시각화** | `recorded_at`과 현재 시각 차이로 **일수 밴드**를 정하고, 밴드별 마커 색·강조를 달리한다(`seed-drop-visual.ts`). 구역 라벨(A/B/C 등)과 병용하여 관제 이력 가독성을 높인다. **블루카본 과학 단계를 단정하지 않고**, 관제·사업 보고용 **UX 정책 상수**로 기술하는 것이 명세 정합성에 유리하다. |

## 4. 기술적 기대 효과 (참고)

- 작업 가능 시간대·살포 한도·회피 항로를 **동일 데이터 파이프라인**에서 참고할 수 있어, 운영 판단의 **재현성**을 높인다.  
- 단말에 DB·공개 지도 키를 두지 않고 **수집 전용 엔드포인트**로만 붙이는 모델로 **노출 면을 줄인다.**  
- 살포·궤적을 **분리 수집·상이한 레이트 리밋**으로 처리하여, 이벤트 버스트와 주기 궤적이 한 병목으로 몰리는 것을 **완화**한다.  
- 살포 점으로 **볼록 껍질 근사 면적**을 산출·표시하여 보고·설명 자료로 활용할 수 있다(정밀 측량 대체 아님).

---

## 5. 변리사 확인·질문 리스트 (상담 시)

1. **청구항 분리:** 계획·한도·구역 점수(포인트 1 일부), A\* **기상가중 스텝 비용**·장애물 회피(포인트 1 일부), 이원 Edge 수집(포인트 2), Hull 면적(포인트 3), 연령 UI(포인트 4)를 **한 출원**으로 묶을지, **통신·계획** 등으로 나눌지.  
2. **A\*·기상:** 스텝 비용의 **풍향·풍속·돌풍 배수**와 **구역·한도·PID**를 **한 독립항**에 묶을지 분리할지, 선행 「해상 경로 + 기상」 조합에 대한 **진보성 방어**를 어떻게 할지.
3. **Convex Hull:** 수학적 일반성 회피를 위해 **입력 정합성(이벤트·GNSS·타임스탬프)·면적 0 처리·투영 근사 한계**를 청구항에서 어떻게 묶을지.  
4. **포인트 4:** 특허 청구 vs 실용신안·디자인·저작권·비공개 중 어느 쪽이 비용 대비 유리한지.  
5. **가출원·지원:** 확보된 UI 캡처·아키텍처 도식만으로 **우선일 확보** 가능 범위와, 이후 정규 명세 보완 시 **실시예 부족** 리스크. IP나래 등은 **당해 연도 공고·자격** 확인 필요.
