# 변리사 검토용 발명 제안서 초안 (코드베이스 정렬본)

> **목적:** 본 저장소에 실제로 존재하는 구현·문서를 근거로, 감정적 확언 없이 발명 요지를 정리한 **내부 초안**입니다. 등록 가능성·청구항 구성·출원 전략은 변리사 판단 및 선행기술 조사 후 확정해야 합니다.  
> **법률 자문 아님:** 본 문서는 특허성 판단이나 침해 여부를 단정하지 않습니다.

---

## 1. 발명의 명칭 (가칭)

**해양 종자 살포 관제를 위한 기상·해역 조건 연동 계획 엔진, 장애물 회피 항로 생성, 단말 이원 수집 게이트웨이 및 살포 면적·연령 시각화를 포함한 시스템**

(영문 병기는 출원 시 변리사와 조정)

---

## 2. 기술 분야

해양 생태 복원(해초대 조성 등)을 위한 **종자 살포 작업 계획·항로·관제·이력 수집** 소프트웨어, 이와 연동되는 **해상 IoT 단말(ESP32 등) 및 서버 측 수집 게이트웨이**에 관한 것.

---

## 3. 배경 기술 및 해결하고자 하는 과제

| 과제 | 내용 (제안 관점) |
|------|------------------|
| 작업 계획의 객관화 | 풍속·파고·시정 등 기상 조건에 따른 작업 한도·시간대 판단, 해역 격자의 점수화가 수동 경험만으로는 일관되기 어렵다. |
| 항로·작업 안전 | 암초·보호구역 등 회피가 필요한 해역에서 살포 목적지까지의 경로를 데이터로 제시할 필요가 있다. |
| 단말 보안·부하 | 지도 API 키·DB 직결 권한을 단말에 두면 유출 위험이 크고, 살포 이벤트와 고빈도 궤적이 한 통로에 섞이면 운영·스케일링이 어렵다. |
| 면적·이력 표현 | 점 단위 살포만으로는 작업 범위를 면적으로 설명하기 어렵고, 시계열이 긴 복원 사업에서는 이력의 **경과 시간**을 화면에서 구분할 필요가 있다. |

## 4. 본 저장소에서 확인되는 구성 (근거 파일)

아래는 **현재 레포에서 읽을 수 있는 구현**을 기준으로 한 것입니다.

### 4.1 기상·작업 계획 연동

- **기상청 Open API 연동(단기·중기 등):** `src/lib/kma-weather.ts` — 공공데이터포털 인증키(`VITE_KMA_SERVICE_KEY` 등) 사용 여부에 따라 실호출·목업 분기.
- **시간대별 작업 가능도·일정 추천과의 연동:** `src/lib/work-recommendation.ts`, `src/app/components/WeatherAIPanel.tsx`, `src/app/WorkPlanView.tsx` 등.
- **기상 스트레스 기반 추정 안착률(휴리스틱):** `src/lib/seeding-day-eval.ts`의 `estimateAttachmentPercent` 등 — 실측 안착률이 아닌 **관제·사후검토용 추정**임이 코드 주석에 명시됨.

### 4.2 해역 구역 점수화 및 살포 한도·PID 제안

- **기상·시뮬 해양 필드(수심·수온·해류 등)를 반영한 구역 랭킹, 일일 살포 한도:** `src/lib/seeding-plan-ai.ts` (`rankSeedingZones`, `calcDailyLimit` 등).
- **Arduino용 PID 파라미터 제안 및 스케치 스니펫 생성:** 동일 파일 및 UI `src/app/components/SeedingPlanPanel.tsx`, `RouteNavPanel.tsx`.

### 4.3 장애물 회피 그리드 A* 항로

- **암초·보호구역 등 장애물 반경 내 셀을 탐색 제외하는 A*:** `src/lib/seeding-plan-ai.ts`의 `findAStarRoute`, `KNOWN_OBSTACLES` — 주석상 향후 KHOA 해도 API 등으로 대체 가능하다고 명시.
- **기상 가중 스텝 비용:** 동일 파일의 `weatherStepCostMultiplier` — 풍향(바람이 오는 방향)과 이동 방위에 따라 역풍일수록 스텝 비용 배수를 키우며, `findAStarRoute(start, goal, wx)`의 세 번째 인자로 `WeatherInput`을 넘기면 적용된다. 생략 시 거리만 사용.
- **구현상 보조:** 휴리스틱 h는 기하학적 거리( haversine )를 사용하므로, 스텝 비용 배수가 항상 ≥1이면 h는 가중 잔여 비용의 하한을 과대평가하지 않도록 설계할 수 있다(구현 세부는 심사·보정 시 검토).

### 4.4 살포 면적 추정 (Convex Hull + 면적 공식)

- **살포 점 집합에 Graham scan 유사 볼록 껍질 후, 위경도 평면 근사로 ha 산출:** `src/lib/seeding-day-eval.ts` (`estimateSeedingAreaHa`, `calcPolygonAreaHa`) — **격자 정밀 측량이 아닌 관제·참고치**임이 주석에 명시됨.
- **관제 UI 표시:** `src/app/Dashboard.tsx` (추정 면적 툴팁 등).

### 4.5 기록 시각 기반 마커 연령·구역 라벨

- **`recordedAt` 기준 일수 밴드(예: 45일·120일·400일·800일 등)에 따른 색상 팔레트:** `src/lib/seed-drop-visual.ts` (`dropAgeBand`, `seedDropMarkerColors` 등).
- **구역 라벨(A/B/C 및 일련번호):** `src/app/Dashboard.tsx` 등 — 지도·이력과 연동.

### 4.6 단말 → 서버: 이원화 수집 게이트웨이

- **살포 이벤트 수집:** `supabase/functions/telemetry-ingest/index.ts` — `X-Device-Ingest-Key`, 페이로드 검증, IP·글로벌 레이트 리밋, `seed_drop_records` upsert.
- **선박 궤적 수집:** `supabase/functions/vessel-track-ingest/index.ts` — 동일 헤더 패턴, `vessel_track_points` 등으로 저장.
- **관제 웹 DB 연동:** `src/lib/marine-db.ts` — `fetchSeedDropRecords`, `fetchVesselTrackPoints`, `insertVesselTrackPoint` 등.
- **하드웨어·연동 설명:** `docs/hardware-arduino-gnss-seed/README.md`, `arduino/README.md`, `arduino/sketches/esp32-marine-telemetry/esp32_marine_telemetry.ino` (주석상 살포·궤적 URL 분리).

### 4.7 제출·설명용 아키텍처 도식 (문서)

- `관공서-제출용/docs/관공서_시스템_아키텍처_구조도.md` — Edge 이원화·검증·레이트 리밋 개념도.

## 5. “청구항 검토 후보”로 옮길 때의 축 (가설, 비단정)

변리사 선행조사 후 **버릴 수 있는 축**까지 포함해 나열합니다.

| 축 | 요지 | 선행기술 검토 시 유의점 |
|----|------|-------------------------|
| A | 기상·시뮬 해양 필드 입력을 받아 **살포 구역 점수화·일일 한도·PID 제안**을 일관된 규칙으로 산출하는 **계획 엔진** | 농업·드론 파종, 일반 항해 지원 SW와의 차별화 논리 필요 |
| B | 암초·보호구역 등 **장애물 회피 제약 하 A* 그리드 항로** 및 웨이포인트화 | A*·해상 경로 자체는 선행 다수. **살포 작업 맥락의 목적함수·입력 데이터 결합**이 논점 |
| C | **살포 이벤트 수집**과 **궤적 수집**을 별도 Edge·스키마로 분리하고, 공통 장비 시크릿·레이트 리밋·검증을 적용하는 **수집 방법** | 일반적인 API 게이트웨이 패턴과의 경계 — **페이로드·한도·DB 역할 분리의 구체 조합·효과**가 논점 |
| D | 살포 점으로 **Convex Hull 근사 면적(ha)** 산출·표시 | Hull·면적 공식 자체는 이론적으로 널리 알려짐 — **입력 정합성(이벤트·좌표)·해상 관제 UI 결합**이 논점 |
| E | `recordedAt` 기반 **연령 밴드 시각화** 및 구역 라벨과의 병용 | UI·정보표시만으로는 보호가 어려울 수 있음 — 실용신안·디자인·영업비밀과 비교 검토 |

## 6. 구현됨 vs 문서·로드맵 (출원 명세 정합성용)

| 항목 | 상태 |
|------|------|
| Edge 이원 수집·검증·레이트 리밋 | **구현됨** (`telemetry-ingest`, `vessel-track-ingest`) |
| Convex Hull 기반 ha 추정 | **구현됨** (`seeding-day-eval.ts`) |
| 기상청 API 연동·작업 가능도 | **구현됨** (키 유무에 따른 분기, `kma-weather.ts` 등) |
| A* 장애물 회피 항로 | **구현됨** (`findAStarRoute`) |
| 기상을 A* 스텝 비용에 반영 | **구현됨** (`weatherStepCostMultiplier`, `findAStarRoute(..., wx)`) |
| 단말 망 단절 시 로컬 큐·재전송 | **문서상 권장** (`arduino/docs/02-해상-통신-범위-한계.md` 등). 기본 스케치가 항상 큐잉을 포함한다고 단정하지 말 것 — **어느 버전 펌웨어에 구현됐는지**를 실시예에 명시 |

## 7. 발명의 효과 (기술·운영 관점, 참고)

- 작업 가능 구간·한도·회피 항로를 **동일한 데이터 파이프라인**에서 참고할 수 있음.
- 단말에 DB·지도 키를 두지 않고 **수집 전용 엔드포인트**로만 붙이는 운영 모델.
- 살포 점으로 **근사 면적**을 산출·표시하여 보고·설명 자료로 활용 가능(정밀 측량 대체 아님).
- 장기 이력에 대해 **시간 경과에 따른 시각 구분**으로 관제 가독성 확보.

## 8. 변리사 상담 시 확인 질문 (예시)

1. 축 A~E 중 **단일 출원**으로 묶는 것과 **분리 출원** 중 어느 쪽이 비용·범위 면에서 유리한가.  
2. 현재 A* 구현(**거리×풍향·풍속·돌풍 배수** + **장애물 회피**)과 **구역·한도·PID**를 **하나의 독립항**으로 쓸 경우, 선행 “해상 경로 + 기상” 조합에 대한 **진보성 방어 전략**은 무엇인가.  
3. Convex Hull 면적·연령 색상 UI는 **특허**로 갈지, **실용신안·디자인·비공개**가 나은지.  
4. 해외 출원·PCT 필요 여부 및, 공공 납품·R&D 과제에서 **명세 공개 수준** 제약이 있는가.
