# 종자 투하(살포) 1건당 신호 전송 — 제안서 (방안 1 · 방안 2)

**목적**: “블루투스로 버튼을 누르는 것”이 아니라, **투하가 발생할 때마다 한 번씩** 관제·DB·모바일 화면에 반영할 수 있는 **현실적인 두 갈래**를 정리한다.  
**전제**: 스마트폰은 **수신기·게이트웨이·표시기** 역할이 될 수 있고, 반드시 “폰 블루투스만”이 정답은 아니다.  
**구현 시 볼 문서(저장소 기준)**: [`docs/hardware-arduino-gnss-seed/README.md`](../hardware-arduino-gnss-seed/README.md) **§7** — `seed_drop_records`, Edge `telemetry-ingest`, 웹 폴링과의 대응을 한곳에 정리했다.

---

## 0. 오해 풀기 — “핸드폰에 블루투스를 장착”하면 뭐가 되나?

- **폰 자체에 추가로 꽂는 USB 블루투스 동글**은, 보통 **PC용**에 가깝고, 안드로이드에서도 “USB OTG + 시리얼”로 쓰는 식이다. **아이폰은 거의 불가**에 가깝다.
- **진짜로 하고 싶은 것**은 대개 다음 중 하나다.  
  1) **거치대(또는 투하기) 안의 모듈**이 **BLE로 숫자/코드를 보내고**, 폰 앱이 받아 **1투하 = 1메시지**로 서버에 올린다.  
  2) **아두이노/ESP32**가 투하를 감지하고 **Wi‑Fi·LTE·USB**로 서버에 올리고, 폰은 **같은 DB를 보기만** 한다.

즉, “블루투스 = 폰에 신호”가 **항상** 정답은 아니다. **투하를 누가 ‘알아채느냐’**(센서·스위치·수동 버튼)가 핵심이다.

---

## 알리익스프레스 등에서 “있나?” — 검색 방향만 (상품 보증 없음)

마켓플레이스는 모델명이 자주 바뀌므로 **키워드·형태**로 찾는 편이 안전하다.

| 찾고 싶은 것 | 검색 예 (영문/한글 혼용) | 비고 |
|--------------|------------------------|------|
| BLE로 소량 데이터 보내기 | `BLE UART module`, `HM-10`, `JDY-08`, `nRF24L01 BLE` | 폰은 **BLE Central**이 되어야 함 (웹은 제한적) |
| 아두이노 + 블루투스 | `ESP32 BLE`, `HC-05 Arduino` (Classic SPP) | HC-05는 **구형**, 안드로이드 위주 |
| “버튼 하나로 무선” | `BLE wireless button`, `BLE panic button`, `iTag clone` | **프로토콜이 비공개**인 경우 많음 → 역공학 필요 |
| USB로 폰에 시리얼 | `USB serial OTG Android`, `CH340 USB UART` | **안드로이드 + OTG** 위주 |

**주의**: “알리에서 파는 BLE 버튼”이 **우리 앱과 바로 호환**된다는 보장은 없다. 대부분 **자사 앱 전용**이거나 **단순 비콘**이다. **MVP는 ‘검증된 BLE UART 모듈 + 자체 펌웨어’**가 낫다.

## 요구사항 정리 — “투하된 게 하나씩 전송”

| 항목 | 권장 |
|------|------|
| 단위 | **투하 1회 = 이벤트 1건** (중복 방지용 시퀀스 번호 SEQ 권장) |
| 페이로드 예 | `{ vessel_id, seq, ts, lat?, lng?, type: "seed_drop" }` |
| 전송 후 | 서버 `seed_drop_records` 또는 Edge Function → **관제 대시보드·/mobile 실시간 반영** |

# 방안 1 — 스마트폰 중심 (거치대 BLE → 폰 → 서버)

**개념**: 거치대(또는 투하기 부착 모듈)가 **BLE로 짧은 프레임**을 보내고, **폰 앱/웹**이 받아 **Supabase에 1건씩 INSERT**한다.

### 구성

- **하드웨어**: 저전력 MCU + BLE 칩(예: ESP32-C3, nRF52 계열) — 투하 감지(미세 스위치·광센서·함정 기존 신호 탭) 시 GATT Write 또는 Notify로 4~16바이트 전송.
- **폰**:  
  - **안드로이드**: Capacitor/React Native + BLE 플러그인 **권장** (안정적).  
  - **웹만**: Web Bluetooth는 **Chrome/Android 일부**에 한정, iOS 불가.
- **서버**: 기존 `insertShipCommand` / `upsertSeedDropRecord` 파이프라인 재사용 또는 전용 API.

### 장점

- 함정에 **이미 있는 폰**만으로 현장 UI(`/mobile`)와 동기 가능.
- 무선으로 **거치대·손쉬운 조작** 가능.

### 단점

- **iOS**는 웹 BLE 불가 → 네이티브 셸 거의 필수.
- BLE 모듈·펌웨어 **직접 개발·인증** 부담.
- 배터리·방수·진동 환경.

### “블루투스로 버튼 누르는 것”과의 관계

- 사용자가 말한 “버튼 누르기”는 **UI를 대신 눌러 주는 HID**가 아니라, **‘투하 이벤트’라는 데이터 한 줄**을 보내는 쪽이 맞다.  
- HID 키보드 에뮬레이션도 가능은 하지만, **오탐·포커스·보안** 때문에 **권장 순위는 낮음**.

# 방안 2 — 아두이노·ESP32 현장 게이트웨이 (폰은 “모니터”만)

**개념**: **ESP32(또는 아두이노+모뎀)** 가 투하를 감지하고 **Wi‑Fi / LTE(NB‑IoT·LTE-M) / LoRa→게이트웨이**로 **바로 서버**에 올린다. 블루투스는 **폰과만** 쓰지 않아도 된다(현장‑클라우드 직결).

### 구성

- **센서/입력**: 함정 투하 스위치·릴레이 접점·ADC(모터 전류 스파이크) 등 **1회 이벤트**로 확정 가능한 신호.
- **통신**:  
  - 항구·연안: Wi‑Fi 또는 선박 라우터.  
  - 원해: **LTE 모듈**(A7670 등)로 HTTPS POST.
- **폰**: `/mobile`은 **DB 구독·폴링**으로 “방금 올라온 투하”만 표시.

### 장점

- **폰 블루투스 스택 의존 최소화** (배터리·백그라운드 이슈 감소).
- **투하 1건 = 1 HTTP/1 MQTT**로 모델이 단순.
- iOS·안드로이드 구분 없이 **관제탑·함정 동시 표시** 용이.

### 단점

- 하드웨어·방수·전원·안테나 설계.
- LTE면 **요금·SIM·인증**.

### “제일 좋은 건 아두이노인가?”

- **아두이노(브랜드)** 자체보다 **ESP32 + 펌웨어** 조합이 실무에서 많다(가격·BLE·Wi‑Fi 동시).  
- **“제일 좋음”**은 조건에 따른다:  
  - **폰만 쓰고 싶다** → 방안 1.  
  - **폰 꺼짐·백그라운드 싫다** → 방안 2.

---

## 방안 1 vs 방안 2 — 요약 비교

| 기준 | 방안 1 (BLE → 폰 → 서버) | 방안 2 (ESP32/아두이노 → 서버) |
|------|---------------------------|--------------------------------|
| 폰 역할 | **필수(수신·전송)** | **선택(표시만)** |
| iOS | 네이티브 권장 | **동일하게 유리** |
| 개발 난이도 | BLE+앱 동기 중간 | 펌웨어+망 중~상 |
| 투하 1건씩 전송 | SEQ 포함 프레임 설계 | **가장 깔끔** |
| 알리 부품 | BLE UART·ESP32 모듈 | ESP32+SIM 모듈 등 |

---

## 권장 로드맵 (두 방안 병행 가능)

1. **PoC**: 방안 2로 **USB 전원 ESP32**를 선실에 두고, 투하 스위치 → **HTTPS 1건** (가장 빨리 “1투하=1전송” 검증).  
2. **현장 UX**: 방안 1로 **거치대 BLE** + 안드로이드 앱에서 수신 → 동일 API.  
3. **운영**: `source: esp32` / `source: ble_phone` 로그 컬럼, SEQ 중복 제거.
