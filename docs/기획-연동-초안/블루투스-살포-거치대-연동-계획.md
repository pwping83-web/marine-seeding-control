# 블루투스 살포 거치대 ↔ 모바일 관제 연동 계획안

**문서 목적**: 함정에서 스마트폰을 거치대에 올려 두고, **물리 버튼(또는 외부 MCU)** 만으로 `살포 시작` / `살포 중지` 등이 **앱의 버튼과 동일하게 동작**하도록 하기 위한 단계별 설계·리스크·검증 항목을 정리한다.  
**현재 구현**: 웹 앱 `/mobile` 화면에 터치 버튼으로 `ship_command_logs`(Supabase) 또는 `BroadcastChannel`(로컬 시연) 전송이 가능하다.  
**투하 1건(살포 점)** 은 별개 파이프라인이다: 함정 화면의 **「투하 (센서 시뮬)」** 는 `seed_drop_records`에 쓰며, 실제 센서는 아두이노 등이 **`telemetry-ingest`** 로 동일 테이블에 넣는 설계가 맞다. 개념·체크리스트는 **[`docs/hardware-arduino-gnss-seed/README.md`](../hardware-arduino-gnss-seed/README.md) §7** 을 본다.

---

## 1. 목표·범위

| 단계 | 목표 |
|------|------|
| **M0 (현재)** | `/mobile` — 지도에 내 위치, 살포 시작/중지, AI 요약, 지금 위험, 금일 살포 건수 |
| **M1** | 거치대에서 **단일 신호**(예: 한 바이트 시리얼)로 시작/중지 **구분** 가능 |
| **M2** | 블루투스(BLE 또는 Classic SPP)로 폰과 페어링 후, 수신 시 **Web API 또는 앱 내 이벤트**와 연동 |
| **M3** | 운영: 페어링·권한·감사 로그·오조작 방지 |

비범위(초기): 해양 무선법규에 따른 VHF 음성 대체, 위성 단독 링크.

## 2. 기술 옵션 비교

### 2.1 Web Bluetooth (Chrome/Android 위주)

- **장점**: 별도 네이티브 앱 없이 웹에서 GATT 연결 가능한 경우가 있음.
- **단점**: iOS Safari **미지원**. 사용자 제스처 후 연결 필요. 장치는 **BLE GATT 서비스** 노출 필요.
- **적합**: 안드로이드 전용 운영, 거치대가 BLE Peripheral.

### 2.2 Capacitor / React Native 래퍼 + 플랫폼 BLE 플러그인

- **장점**: iOS·Android 공통으로 **시리얼/BLE** 브리지 가능.
- **단점**: 스토어 배포·빌드 파이프라인 추가.
- **적합**: 함정 표준이 iOS 포함일 때 중기 권장.

### 2.3 거치대 = USB OTG 키보드 HID 에뮬레이션

- MCU가 **미디어 키 또는 조합키**를 보내고, 웹은 `keydown`으로 감지(포그라운드 필요).
- **장점**: 브라우저 BLE 불필요.
- **단점**: OS 포커스·접근성 충돌, 키 매핑 관리.

### 2.4 거치대 → LTE/LoRa → 서버 → Supabase `ship_command_logs`

- 함정 폰은 구독만 하고, **물리 버튼은 망을 통해 직접 DB**에 쓰기.
- **장점**: 폰 블루투스 스택 의존 감소.
- **단점**: 인증·Replay 방지·망 비용.

**권장 혼합**: M1은 **HID 또는 단순 UART → OTG 시리얼 리더(안드로이드)** 로 빠른 파일럿, M2에서 **Capacitor + BLE** 로 iOS까지 확장.

## 3. 거치대 펌웨어(프로토콜) 초안

### 3.1 단일 채널 시리얼(예: 9600 8N1)

| 바이트(또는 프레임) | 의미 | 앱 동작 |
|---------------------|------|---------|
| `0xA1` | 살포 시작 | `insertShipCommand(..., 'seed_start')` 와 동일 |
| `0xA2` | 살포 중지 | `seed_stop` |
| `0xA3` | 위치 보고(옵션) | 추후 `report_position` 과 동일 파이프라인 |

**보안**: 고정 1바이트만 쓰면 오인식 위험 → **프레임** `[0x55, CMD, SEQ, XOR]` 4바이트 이상 권장.

### 3.2 BLE GATT

- **Service UUID**(예비): 벤더 할당 128-bit UUID 1개.
- **Characteristic `command`**: Write Without Response, 1~20 byte.
- **Characteristic `ack`**(선택): Notify로 수신 확인.

## 4. 모바일 앱 쪽 작업 항목

1. **수신 레이어**: `navigator.serial` (Chrome, USB) / Web Bluetooth / 네이티브 플러그인 중 택1.
2. **디바운스**: 200ms 내 중복 CMD 무시.
3. **상태 동기**: 이미 `subscribeShipCommandInserts` 로 관제탑과 동기 — 거치대에서 온 CMD도 **동일 테이블**에 쓰면 탑·모바일·함정 일치.
4. **UI**: 수신 시 햅틱·토스트(`살포 시작 신호 수신`).
5. **오프라인**: DB 불가 시 현재와 같이 `BroadcastChannel` 한계 명시 → **오프라인 큐**(IndexedDB) 후 재전송 옵션.

---

## 5. 보안·운영

- **페어링 PIN** 또는 버튼 Long-press 학습 모드.
- **선박 ID**(`VITE_VESSEL_LTE_ID`)와 CMD에 **vessel_id** 일치 검증(서버 RLS·Edge Function).
- **감사**: `ship_command_logs`에 `source: 'bt_dock'` 컬럼 추가 검토.
- **Replay**: SEQ 단조 증가 + 서버측 최소 간격 제한.

## 6. 일정(예시)

| 주차 | 산출물 |
|------|--------|
| W1~2 | 프로토콜 확정(프레임·CMD 표), 목업 MCU 펌웨어 |
| W3 | 안드로이드 + USB 시리얼 또는 BLE POC → `/mobile` 에서 CMD 수신 시 `sendCmd` 호출 |
| W4 | 함정 시험(해상 GPS·손망 실측), 오조작·지연 측정 |
| W5+ | iOS 필요 시 Capacitor 이관, RLS·source 필드, 운영 매뉴얼 |
